# XXXIII. Multiple Tables - Introduction

## 1. Why Do We Need Multiple Tables?

When there is too much data, storing everything in a single table can cause:

* Data to be repeated many times.
* Wasted storage space.
* Difficult management.
* Difficult updates.

## 2. Example

Suppose a magazine company has the following data:

| order_id | customer_name | address  | subscription_description | price |
| -------- | ------------- | -------- | ------------------------ | ----- |
| 1        | John          | New York | Sports Magazine          | 10    |
| 2        | John          | New York | Fashion Magazine         | 15    |
| 3        | Mary          | Texas    | Sports Magazine          | 10    |

### Problem

John's name and address appear multiple times:

```text
John - New York
John - New York
```

Subscription names are also repeated:

```text
Sports Magazine
Sports Magazine
```

If the table contains millions of rows, this becomes very inefficient.

## 3. Solution

Split the data into multiple tables.

### customers table

Stores customer information.

| customer_id | customer_name | address  |
| ----------- | ------------- | -------- |
| 1           | John          | New York |
| 2           | Mary          | Texas    |

### subscriptions table

Stores subscription information.

| subscription_id | description      | price_per_month |
| --------------- | ---------------- | --------------- |
| 1               | Sports Magazine  | 10              |
| 2               | Fashion Magazine | 15              |

### orders table

Stores order information.

| order_id | customer_id | subscription_id | purchase_date |
| -------- | ----------- | --------------- | ------------- |
| 1        | 1           | 1               | 2024-01-01    |
| 2        | 1           | 2               | 2024-02-01    |
| 3        | 2           | 1               | 2024-03-01    |

## 4. Role of Each Table

### customers

Contains:

Customer information

Columns:

* customer_id
* customer_name
* address

### subscriptions

Contains:

Subscription information

Columns:

* subscription_id
* description
* price_per_month
* subscription_length

### orders

Contains:

Order information

Columns:

* order_id
* customer_id
* subscription_id
* purchase_date

JOIN is used to combine data from multiple tables.

Simply put:

**JOIN = Merge tables together.**


## 3. Basic Syntax

```sql

FROM table1
JOIN table2
ON table1.column_name = table2.column_name;
```

**Step 1**

**Step 2**


Add the `customers` table.

**Step 3**


| order_id | customer_id |
| -------- | ----------- |

**customers**

# XXXVI. Inner Joins

## What is an Inner Join?

In the previous lesson, we used:

```sql
SELECT *
FROM orders
JOIN customers
ON orders.customer_id = customers.customer_id;
```

In fact:

```sql
JOIN
```

is the same as:

```sql
INNER JOIN
```

The following two statements are identical:

```sql
SELECT *
FROM orders
JOIN customers
ON orders.customer_id = customers.customer_id;
```

```sql
SELECT *
FROM orders
INNER JOIN customers
ON orders.customer_id = customers.customer_id;
```


## How Does an Inner Join Work?

An Inner Join keeps only the rows that have matching data in both tables.

Example:

**customers**

| id | name  |
| -- | ----- |
| 1  | John  |
| 2  | Mary  |
| 3  | David |

**orders**

| order_id | customer_id |
| -------- | ----------- |
| 101      | 1           |
| 102      | 2           |
| 103      | 5           |

Query:

```sql
SELECT *
FROM orders
JOIN customers
ON orders.customer_id = customers.id;
```

SQL checks:

| customer_id | Exists in customers? |
| ----------- | -------------------- |
| 1           | Yes                  |
| 2           | Yes                  |
| 5           | No                   |

Result:

| order_id | customer_id | name |
| -------- | ----------- | ---- |
| 101      | 1           | John |
| 102      | 2           | Mary |

The row with:

```text
customer_id = 5
```

is removed because there is no matching customer in the `customers` table.


## Rule of an Inner Join

An Inner Join only returns:

* Rows that satisfy the `ON` condition.

Rows that do not match are excluded from the result.

## Key Takeaways

**COUNT(*)** – Counts rows.

```sql
SELECT COUNT(*)
FROM table_name;
```

**INNER JOIN** – Combines two tables.


## Lesson Summary

# XXXVII. Left Join

A **LEFT JOIN** keeps all rows from the left table, even when there is no matching row in the right table.

## Syntax

```sql
SELECT *
FROM table1
LEFT JOIN table2
ON table1.id = table2.id;
```


## Example

**newspaper**

| id |
| -- |
| 1  |
| 2  |
| 3  |

**online**

| id |
| -- |
| 2  |
| 3  |

Query:

```sql
SELECT *
FROM newspaper
LEFT JOIN online
ON newspaper.id = online.id;
```

Result:

| newspaper.id | online.id |
| ------------ | --------- |
| 1            | NULL      |
| 2            | 2         |
| 3            | 3         |

The row with `id = 1` is still included even though it does not exist in the `online` table.


## IS NULL

Used to find rows that do not have matching data.

```sql
WHERE online.id IS NULL
```


## Exercise 1

Join `newspaper` and `online`.

```sql
SELECT *
FROM newspaper
LEFT JOIN online
ON newspaper.id = online.id;
```


## Exercise 2

Find people who subscribe to the print newspaper but not the online newspaper.

```sql
SELECT *
FROM newspaper
LEFT JOIN online
ON newspaper.id = online.id
WHERE online.id IS NULL;
```


## Remember

| INNER JOIN                          | LEFT JOIN                                   |
| ----------------------------------- | ------------------------------------------- |
| Only matching rows                  | Keeps all rows from the left table          |
| Non-matching rows are removed       | Non-matching rows are preserved             |
| No NULL values from the right table | NULL values may appear from the right table |


## Lesson Summary

* LEFT JOIN keeps all data from the left table.
* If no matching row exists in the right table, SQL fills the right-side columns with `NULL`.
* `WHERE ... IS NULL` is commonly used to find records that exist in the left table but not in the right table.

# XXXVIII. Primary Key vs Foreign Key

## 1. What is a Primary Key?

A **Primary Key** is a column used to uniquely identify each row in a table.

Example:

**customers**

| customer_id | name    |
| ----------- | ------- |
| 1           | Alice   |
| 2           | Bob     |
| 3           | Charlie |

Here, **customer_id** is the Primary Key because every customer has a unique ID.


## 2. Requirements of a Primary Key

### Cannot be NULL

Incorrect:

| customer_id | name  |
| ----------- | ----- |
| NULL        | Alice |


### Cannot contain duplicate values

Incorrect:

| customer_id | name  |
| ----------- | ----- |
| 1           | Alice |
| 1           | Bob   |


### Each table can have only one Primary Key

Example:

`customers.customer_id`

is the Primary Key of the **customers** table.


## 3. What is a Foreign Key?

A **Foreign Key** is a Primary Key from one table that appears in another table.

Example:

**customers**

| customer_id | name    |
| ----------- | ------- |
| 1           | Alice   |
| 2           | Bob     |
| 3           | Charlie |

Primary Key: `customer_id`

**orders**

| order_id | customer_id |
| -------- | ----------- |
| 1        | 2           |
| 2        | 2           |
| 3        | 3           |

In the **orders** table, `customer_id` is no longer a Primary Key.

It is a **Foreign Key** because it references:

`customers.customer_id`


## 4. Easy Way to Remember

In the original table:

`customers.customer_id`

→ **Primary Key**

When it appears in another table:

`orders.customer_id`

→ **Foreign Key**


## 5. Why Do We Need Foreign Keys?

Foreign Keys allow tables to be connected.

Example:

**orders**

| order_id | customer_id |
| -------- | ----------- |
| 1        | 2           |
| 2        | 2           |
| 3        | 3           |

We know which customer placed each order.

To find the customer's name, we must connect this table with the **customers** table.


## 6. JOIN Usually Uses a Primary Key and a Foreign Key

Example:

```sql
SELECT *
FROM orders
JOIN customers
ON orders.customer_id = customers.customer_id;
```

Here:

* `orders.customer_id` is a Foreign Key.
* `customers.customer_id` is a Primary Key.


## 7. Exercise

**classes**

Primary Key:

`id`

**students**

Primary Key:

`id`

Foreign Key:

`class_id`

because it references:

`classes.id`


## 8. Solution

```sql
SELECT *
FROM students
INNER JOIN classes
ON students.class_id = classes.id;
```


## Summary

| Concept     | Meaning                                         |
| ----------- | ----------------------------------------------- |
| Primary Key | Uniquely identifies each row                    |
| Foreign Key | References the Primary Key of another table     |
| Primary Key | UNIQUE + NOT NULL                               |
| JOIN        | Usually connects a Foreign Key to a Primary Key |

Quick formula:

**Primary Key = The ID of its own table**

**Foreign Key = An ID copied from another table**
**to create a relationship between tables**

Most SQL JOIN statements are based on:

**Foreign Key = Primary Key**


# XXXIX. Cross Join

## 1. What is CROSS JOIN?

**CROSS JOIN** combines every row from Table A with every row from Table B.

Simply put:

Each row in Table A is paired with every row in Table B.


## 2. Syntax

```sql
SELECT *
FROM table1
CROSS JOIN table2;
```

Unlike **INNER JOIN**:

```sql
SELECT *
FROM table1
INNER JOIN table2
ON table1.id = table2.id;
```

CROSS JOIN:

* Does not require an `ON` clause.
* Does not require a join condition.

because it combines everything with everything.


## 3. Example

**shirts**

| shirt_color |
| ----------- |
| white       |
| grey        |
| olive       |

**pants**

| pants_color |
| ----------- |
| light denim |
| black       |

Query:

```sql
SELECT shirts.shirt_color,
       pants.pants_color
FROM shirts
CROSS JOIN pants;
```

Result:

| shirt_color | pants_color |
| ----------- | ----------- |
| white       | light denim |
| white       | black       |
| grey        | light denim |
| grey        | black       |
| olive       | light denim |
| olive       | black       |


## 4. Number of Rows After a CROSS JOIN

Formula:

**Rows = Rows in Table A × Rows in Table B**

Example:

3 shirts × 2 pants = 6 rows

If:

10 rows × 12 rows = 120 rows


## 5. When is CROSS JOIN Used?

It is rarely used for matching shirts and pants.

More commonly, it is used when:

* Comparing each row against a list of values.
* Generating combinations such as:

  * 12 months of the year
  * 7 days of the week
  * Regions
  * Status values


## Summary

| JOIN       | Meaning                      |
| ---------- | ---------------------------- |
| INNER JOIN | Only matching rows           |
| LEFT JOIN  | All rows from the left table |
| CROSS JOIN | Every row with every row     |

Quick reminder:

**INNER JOIN = Match related rows**

**CROSS JOIN = Match everything with everything**

**Rows = A × B**

For this lesson:

```
newspaper × months
        ↓
Create every customer-month pair
        ↓
Filter active subscription months
        ↓
Count subscribers per month
```


# XL. UNION

## 1. What is UNION?

**UNION** combines the results of multiple SELECT statements into a single result set.

Simply put:

**UNION = Stack one table on top of another**


## 2. Syntax

```sql
SELECT *
FROM table1

UNION

SELECT *
FROM table2;
```

Result:

Data from `table1`

*

Data from `table2`

| ---------- | ----- |
| Bulbasaur  | Grass |
| Charmander | Fire  |
| Squirtle   | Water |

| ------- | ------ |

```sql
SELECT *
FROM table1

FROM table2;
```

| pokemon    | type   |
| ---------- | ------ |
| Bulbasaur  | Grass  |
| Charmander | Fire   |
| Squirtle   | Water  |
| Snorlax    | Normal |


## 4. Requirements for UNION

### Tables must have the same number of columns

Correct:

**table1**

| name | age |
| ---- | --- |
| Tom  | 20  |

**table2**

| name | age |
| ---- | --- |
| Anna | 21  |

Incorrect:

**table1**

| name | age |

**table2**

| name | age | city |

Different number of columns.


### Data types must correspond

Correct:

| name | age |
| ---- | --- |
| Tom  | 20  |

| name | age |
| ---- | --- |
| Anna | 21  |

Incorrect:

| name | age |
| ---- | --- |
| Tom  | 20  |

| city | salary |
| ---- | ------ |
| HCM  | 5000   |

Column order and data types do not match.


## 5. UNION vs JOIN

Combines data horizontally.

→ Adds columns

Example:

### UNION

Combines data vertically.

Table A

Table B

↓

Stack together

Example:

| name |
| ---- |

| Anna |
| John |
| Lisa |

# XLI. WITH

## 1. What is WITH?

**WITH** is used to create a temporary table from the result of a query.

Simply put:

**WITH = Give a name to a query result**
so it can be reused like a normal table.


## 2. Why Use WITH?

Suppose we want to know:

How many subscriptions each customer has.

We can write:

```sql
SELECT customer_id,
       COUNT(subscription_id) AS subscriptions
FROM orders
GROUP BY customer_id;
```

Result:

| customer_id | subscriptions |
| ----------- | ------------- |
| 1           | 2             |
| 2           | 5             |
| 3           | 1             |

However, the marketing team does not care about:

* customer_id = 1
* customer_id = 2
* customer_id = 3

They want to see:

* Alice
* Bob
* Charlie

The customer names are stored in:

**customers**

| customer_id | customer_name |
| ----------- | ------------- |
| 1           | Alice         |
| 2           | Bob           |
| 3           | Charlie       |

We need:

**Step 1**

Calculate subscriptions per customer.

**Step 2**

JOIN with the customers table.

WITH makes this easier.


## 3. Syntax

```sql
WITH temp_table_name AS (
    SELECT ...
)
SELECT *
FROM temp_table_name;
```

Example:

```sql
WITH previous_query AS (
    SELECT customer_id,
           COUNT(subscription_id) AS subscriptions
    FROM orders
    GROUP BY customer_id
)
SELECT *
FROM previous_query;
```


## 4. Simple Explanation

SQL will:

**Step 1**

Run the query inside WITH:

```sql
SELECT customer_id,
       COUNT(subscription_id) AS subscriptions
FROM orders
GROUP BY customer_id;
```

Result:

| customer_id | subscriptions |
| ----------- | ------------- |
| 1           | 2             |
| 2           | 5             |
| 3           | 1             |


**Step 2**

Temporarily save the result as:

`previous_query`


**Step 3**

Use it like a real table:

```sql
FROM previous_query
```


## 5. Exercise

Given query:

```sql
SELECT customer_id,
       COUNT(subscription_id) AS subscriptions
FROM orders
GROUP BY customer_id;
```

Task:

Put the query inside a WITH clause.

Name it:

`previous_query`

Then JOIN it with:

`customers`

and select:

* `customers.customer_name`
* `previous_query.subscriptions`


## Solution

```sql
WITH previous_query AS (
    SELECT customer_id,
           COUNT(subscription_id) AS subscriptions
    FROM orders
    GROUP BY customer_id
)
SELECT customers.customer_name,
       previous_query.subscriptions
FROM previous_query
JOIN customers
ON previous_query.customer_id = customers.customer_id;
```


## Explanation

### Inside WITH

```sql
SELECT customer_id,
       COUNT(subscription_id) AS subscriptions
FROM orders
GROUP BY customer_id;
```

Calculates the number of subscriptions for each customer.


### JOIN

```sql
FROM previous_query
JOIN customers
```

Combines the temporary table with the customers table.


### JOIN Condition

```sql
ON previous_query.customer_id = customers.customer_id
```

Connects:

**Foreign Key ↔ Primary Key**


### SELECT

```sql
SELECT customers.customer_name,
       previous_query.subscriptions
```

Result:

| customer_name | subscriptions |
| ------------- | ------------- |
| Alice         | 2             |
| Bob           | 5             |
| Charlie       | 1             |


## 6. When Should You Use WITH?

When a query becomes long or contains multiple steps.

Instead of:

```sql
SELECT ...
FROM (
      SELECT ...
      FROM ...
     ) AS temp
JOIN ...
```

you can write:

```sql
WITH temp AS (
    SELECT ...
)
SELECT ...
FROM temp;
```

which is much easier to read.


## Summary

| Component      | Meaning                                    |
| -------------- | ------------------------------------------ |
| WITH           | Creates a temporary table                  |
| AS (...)       | Query that builds the temporary table      |
| previous_query | Temporary table name                       |
| JOIN           | Uses the temporary table like a real table |

Quick reminder:

```
WITH
  ↓
Run a query first
  ↓
Store the result temporarily
  ↓
Give it a name
  ↓
Continue using SELECT or JOIN normally
```

General pattern:

```sql
WITH temp_table_name AS (
    SELECT ...
)
SELECT ...
FROM temp_table_name;
```

This is also known as a **CTE (Common Table Expression)** and is widely used in real-world SQL because it makes queries easier to read, maintain, and debug.
